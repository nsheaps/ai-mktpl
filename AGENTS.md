## Key Files & Their Purpose

### Claude Code Configuration

- **.claude/settings.json**: Project-level Claude Code settings with SessionStart hooks (checked into git)
- **.claude/settings.local.json**: Personal user settings (gitignored, not tracked)
- **.claude/hooks/session-start.sh**: Auto-execution script for web sessions - installs dependencies, loads context

### Marketplace Files

- **.claude-plugin/marketplace.json**: Central registry of all plugins. **Auto-updated by CD workflow** - DO NOT manually edit unless necessary
- **plugins/*/вЂ‹.claude-plugin/plugin.json**: Individual plugin metadata including version numbers

### Linting Configuration

- **.markdownlint-cli2.jsonc**: Markdown linting rules (line length: 300, disabled strict rules for docs)

### CI/CD Workflows

#### ci.yaml (Continuous Integration)

- **Triggers**: PR to any branch, push to main, manual dispatch
- **Jobs**:
  - `lint`: Multi-language linting with auto-fix
    - Runs linters for YAML, JSON, Markdown, TypeScript, JavaScript, Python
    - Auto-fixes issues and pushes back to branch
    - Removes old lint error comments on PRs
    - Reruns after fixes to capture unfixable errors
    - Comments unfixable errors on PR using peter-evans actions
    - **ALWAYS FAILS** if any lint errors found (even after auto-fix)
  - `validate`: Validates plugin structure and marketplace consistency

#### cd.yaml (Continuous Deployment)

- **Triggers**: Push to main or PR with `plugins/**` changes
- **Jobs**:
  - `check-version-bump`: Enforces semantic versioning in PRs
    - Uses `svu` utility for version suggestions
    - **FAILS** if plugin modified without version bump
    - Every plugin change requires a version bump - no exceptions
  - `update-marketplace`: Updates marketplace.json on main
    - Syncs plugin versions from plugin.json files
    - Commits with `[skip ci]` marker
    - Generates Homebrew formula for distribution

#### review.yaml (Code Review)

- **Triggers**: PR opened/synchronized/reopened
- **Jobs**: AI-powered code review using Claude Code

### Important Conventions

1. **Version Bumps**: REQUIRED for every plugin change
   - Patch (x.y.Z): Bug fixes, non-breaking changes
   - Minor (x.Y.0): New features, backwards compatible
   - Major (X.0.0): Breaking changes

2. **Commit Messages**: Use conventional commits format
   - `feat:` - New features
   - `fix:` - Bug fixes
   - `chore:` - Maintenance tasks
   - `docs:` - Documentation updates
   - Append `[skip ci]` to skip CI workflows when needed

3. **Git Pushes**: All workflows use retry logic (4 attempts, exponential backoff: 2s, 4s, 8s, 16s)

4. **Secrets**:
   - `GITHUB_PAT_TOKEN_NSHEAPS`: Personal access token for authenticated pushes
   - `ANTHROPIC_API_KEY`: For Claude Code review workflow

## Plugin Development Guidelines

### Technology Preferences

**For compiled binaries and complex plugins:**

- вњ… **Prefer Bun** as the runtime (fast, TypeScript-native, all-in-one tooling)
- вњ… **Prefer TypeScript** for type safety and better developer experience
- вњ… Use `bun build --compile` for standalone executables
- вњ… TypeScript provides better IDE support and catches errors early

**Why Bun + TypeScript:**

- Single tool for package management, building, testing, and running
- Native TypeScript support (no transpilation step needed)
- Fast startup and execution times
- Can compile to standalone binaries with `bun build --compile`
- Better than Node.js for CLI tools and automation scripts

### Adding a New Plugin

1. Create plugin directory: `plugins/plugin-name/`
2. Add `.claude-plugin/plugin.json` with metadata
3. Add plugin content (commands/ or skills/)
4. For complex plugins: Use TypeScript + Bun for compiled binaries
5. Add README.md with documentation
6. **Bump version** in plugin.json (start at 1.0.0)
7. Create PR - CI will validate structure
8. On merge to main, CD workflow updates marketplace.json automatically

### Plugin Structure Requirements

Every plugin must have:

- `.claude-plugin/plugin.json` with required fields:
  - `name`: Plugin identifier (lowercase, hyphen-separated)
  - `version`: Semver version (x.y.z)
  - `description`: Clear, concise description
  - `author`: Name and email
- README.md with usage instructions
- Either `commands/` or `skills/` directory (or both)

### Version Management

- **In PRs**: CD workflow checks that plugin versions are bumped
- **On main**: CD workflow syncs versions to marketplace.json
- **Never** manually edit marketplace.json versions - they're auto-synced from plugin.json files

## Workflow Behavior

### When You Open a PR

1. CI lints all files, auto-fixes issues, pushes fixes
2. CI validates plugin structure
3. CD checks version bumps if plugins changed
4. Review workflow runs Claude Code review
5. Lint errors commented on PR if unfixable

### When You Push to Main

1. CI runs validation
2. CD updates marketplace.json if plugins changed
3. CD generates/updates Homebrew formula
4. Both commit with `[skip ci]` to avoid loops
